# Capabilities vs Skills: Architecting Knowledge Systems

## The Paradigm Shift

Business schools have traditionally taught students to use technology—Excel formulas, Python scripts, BI dashboards. These are **skills**: discrete, tool-specific competencies that enable immediate productivity. But as AI systems like Claude Skills, GPT Actions, and custom agents reveal what employers actually need, a different picture emerges: graduates who can **architect capabilities**—designing reusable, composable knowledge systems that encode business logic and orchestrate intelligent workflows.

This distinction aligns directly with A²I's founding insight:
> "Capability is cheap; coherence is scarce."

When AI can be "taught" domain expertise through structured skill packages, the question shifts from *what tools can you use?* to *what systems can you design, govern, and orchestrate?*

---

## Defining the Distinction

| Dimension | Skills | Capabilities |
|-----------|--------|--------------|
| **Focus** | Tool fluency and task execution | System architecture and orchestration |
| **Question** | "Can you write this Python script?" | "Can you design a workflow where AI, humans, and data deliver verifiable value?" |
| **Durability** | Decays as tools evolve | Endures across technological shifts |
| **Scope** | Individual productivity | Organizational and societal impact |
| **Assessment** | Correct outputs, code that runs | Coherent systems, governance integrity, stakeholder alignment |
| **Example** | Building an Excel dashboard | Architecting a data ecosystem with observability, feedback loops, and trust mechanisms |

---

## AI Systems as a Case Study

Modern AI platforms demonstrate this shift vividly:

### Claude Skills & Custom Agents
- **Skills approach:** Teach students to prompt Claude effectively for individual tasks.
- **Capabilities approach:** Teach students to architect reusable skill packages—modular knowledge systems that encode domain expertise, maintain context, and compose into larger workflows.

### Implications for Education
When AI systems can be configured with domain knowledge, business logic, and governance constraints, graduates must be able to:
1. **Frame problems** worthy of intelligent automation.
2. **Design knowledge structures** that AI agents can execute reliably.
3. **Orchestrate human–AI collaboration** with clear accountability.
4. **Govern outcomes** through observability and reflexive improvement.
5. **Adapt architectures** as missions and technologies evolve.

These are not Python skills or Excel skills—they are **system orchestration capabilities**.

The A²I curriculum responds to this reality by teaching students to become **system orchestrators** rather than tool users.

## Teaching Capability Architecture: Practical Moves

### For Faculty
1. **Frame assignments as system design challenges**, not coding exercises.
   - Instead of: "Build a sentiment analysis model."
   - Try: "Design a feedback system where AI synthesizes customer sentiment, humans interpret strategic implications, and dashboards maintain observability."

2. **Assess architectures, not outputs.**
   - Oral vivas should probe: Why this workflow? What breaks if this component fails? How does governance scale?

3. **Use AI tools as design materials**, not teaching topics.
   - Students should configure AI agents to embody business logic, then critique whether the architecture delivers coherent value.

### For Students
1. **Think in systems, not scripts.**
   - Every project should answer: Who does what? Where is trust verified? How does learning happen?

2. **Document design rationale.**
   - Code alone is insufficient. Explain why this architecture, what assumptions it encodes, and how it adapts.

3. **Practice composability.**
   - Build modules that can be reused, combined, and governed—not monolithic solutions.

### Empowerment ↔ Plasticity

### Cyber-Social Learning

### AI Feedback and Assessment
